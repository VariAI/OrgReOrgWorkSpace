# 需求:第一版 MVP 边界

> 讨论:`../discussions/2026-07-23-product-definition.md`;产品形态:`../design/app-overview.md`。
> **下面的 scope 是提议,待用户圈定。**

## 目标

最小证明「agent 里的 IM 为每个员工统一工作、为管理者提供决策」成立——**dogfood 变分小组自己**(绕开换 IM 门槛)。

## 先服务谁

变分小组自己团队(你们已在用这套 IM)。

## MVP 最小闭环(提议)

在自有 IM 里跑通一条闭环:

1. 员工在群聊里正常工作;
2. **数字员工**从群聊识别出建议(先聚焦三类:**任务 + 风险 + 交付物**);
3. 员工**确认**后 → 进「群看板」+「统一工作台」;**回写**群里;可溯源到源消息;
4. **管理者**打开工作台能看到跨项目的**任务 / 进展 / 风险** rollup(且只含其有权看的)。

## In Scope(提议)

- 消息(IM 群聊)+ 群看板(工作目标 + 完成标准 + 待处理 + 风险与阻塞)。
- 统一工作台:统一行动收件箱(task/risk/review)+ 我的协作群(含风险汇总)。
- 数字员工:每群 1 个主要数字员工,识别 → 建议 → 人确认;权限 = 按群授权、只建议。
- 工作主题聚合:**先只做「项目」flavor**(不做故障/部门)。
- 复用平台已有合成 / 分类 / 投影能力,投影进新 IM。

## Out of Scope(一期)

- 对外多租户 / 卖企业(A 模型的对外落地,后续)。
- 身份统一映射细节(已 deferred:`../plans/2026-07-23-identity-unification.md`)。
- 连第三方 IM(= 被否的 B)。
- 12 个外部系统集成(先 0~1 个,或纯 IM 源)。
- 日程智能建议、决策库、企业 AI 统一入口(提问/搜索/交办)、管理后台治理——二期。

## 开放项(记录,不阻塞)

- 项目/工作主题如何产生:声明式 vs 数字员工自动发现。
- 跨群按项目归类的精度阈值。
- MVP 到底切多小(上面是提议,待用户圈定 In Scope)。

## 验收(提议)

- 一条群聊消息 → 数字员工建议 → 人确认 → 出现在群看板 + 工作台 + 回写群 + 可跳回源消息。
- 管理者工作台能看到该项目的任务/进展/风险,且只含其有权访问的。

## 验证

- 子仓实现后各自验证命令(见 `../context/project-context.md`);dogfood 真实使用冒烟。
- 本层只出需求 + 验收,不写子仓代码。
