# 需求:身份统一(IM ⟷ 上下文平台)

## 目标

员工在 IM 登录 = 进入自己的 agent 工作区。IM 用户与平台的组织/人员身份是**同一个自然人**,一套身份、一次认证。这是「agent 里的 IM」(L2)的地基:**平台拥有身份,IM 只是渲染**。

## 背景 / 现状(据各仓,实施前需核实)

- IM(OrgReOrgServer)有自己的用户体系(手机号登录等),Admin 管理这些账号。
- 上下文平台(OrgReOrg)有组织/人员注册表 `workspaces/<person-id>/`(person-id 历史 = 企微 userid)。
- 两套身份目前无映射。

## In Scope

- IM 账号 ⟷ 平台人员一一映射(同一自然人)。
- 单点认证:IM 登录态可换取/校验平台身份(或同源签发)。
- 员工登录 IM 后,客户端能拿到「我是平台里的谁」。

## Out of Scope(一期)

- 权限投影的细粒度模型(后续缝)。
- 组织架构同步(另一根缝)。
- 旧连接器(Matrix/企微)迁移退役本身(单独任务)。

## 用户与角色

- 企业员工:一次登录,IM + agent 同一身份。
- 管理员:在 Admin 管理的账号 = 平台里的人员。

## 主要流程

1. 员工在 IM 客户端登录 → Server 认证 → 映射/换取平台人员身份 → 客户端持有统一身份。
2. 新员工在 Admin 建号 → 同步/触发平台人员注册(person workspace)。

## 业务规则

- 一个自然人在两侧只对应一个身份(唯一性,防重复人格)。
- 身份映射事实源单一(以谁为准 → 见 plan OQ)。

## 数据 / 模型影响(跨仓)

- 需要 IM userid ⟷ platform person-id 的映射(位置/所有权待定)。
- 可能只读消费 Server 用户模型(不改其内部,除非 plan 另定)。

## API / 集成影响

- Server 需暴露「用登录态换/校验平台身份」接口(契约,受保护区)。
- 平台需接受 IM 身份并解析到 person。

## 验收标准

- 同一员工在 IM 与平台被解析为同一 person-id。
- 新建 IM 账号后,平台侧出现对应人员(或反之),无重复。
- 冒烟:一个合成测试员工登录 IM → 平台能回答「这是谁 + 其工作区」。

## 验证备注

- 依赖 plan 首步的两侧身份模型调研。
- 真实凭据/组织数据不进 git;用合成账号验证。
